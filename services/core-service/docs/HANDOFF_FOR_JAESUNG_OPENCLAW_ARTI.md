## 0) 한 줄 요약

`soom-church`는 **단일 관리자 데모 앱**에서 **교회별 워크스페이스를 갖는 SaaS 플랫폼 구조**로 빠르게 전환 중이며,
현재는 **브랜딩(숨=플랫폼) / 멀티테넌트 스키마 / 워크스페이스 접근 가드 / 데모 사용자-플랫폼 관리자 분리 / 공홈 브랜드 랜딩**까지 1차적으로 정리됨.

---

## 1) 제품 정체성(매우 중요)

### 확정된 정체성
- **숨(SOOM)** = 제품/플랫폼/브랜드
- **대흥교회 이음두빛** = 숨 위에서 동작하는 데모 교회 워크스페이스(인스턴스)

### 절대 헷갈리면 안 되는 것
- 숨을 교회명처럼 취급하지 않기
- 데모 교회를 숨 자체처럼 노출하지 않기
- 홈에서 데모 계정/데모 교회를 전면 홍보하지 않기

### UI/문구 위계
- 플랫폼 레벨: SOOM PLATFORM
- 워크스페이스 레벨: [교회명] 워크스페이스

## 2) 현재까지 작업 흐름 요약 (타임라인식)

## A. 초기 빌드 / 운영 허브 MVP
- Next.js + TS + Tailwind + Prisma + PostgreSQL 기반 초기 구축
- 핵심 화면 구현:
  - 로그인
  - 대시보드
  - 교인(목록/상세/등록/수정)
  - 교구/목장
  - 신청
  - 공지
  - 활동로그
- 운영 대시보드 중심(후속관리, 빠른작업, 요약 위젯 등)

## B. 모바일/UX 반복 개선
- 모바일 드로어, active 메뉴, 카드형 리스트, 후속관리 우선순위 개선
- 레이아웃 깨짐(드로어 열 때 본문 압축) 이슈 해결
- 핵심 목표: 모바일에서도 “10초 안에 해야 할 일 파악”

## C. SaaS 전환 1차 (구조 울타리)
- Prisma 멀티테넌트 모델 추가:
  - Church
  - ChurchMembership
  - Subscription (+ plan/status enum)
- 핵심 운영 엔티티 churchId 스코프 추가
- 라우팅 분리:
  - 공홈: `/`, `/features`, `/pricing`, `/contact`, `/login`, `/signup`
  - 앱: `/app/[churchSlug]/*`

## D. SaaS 전환 2차 (실동선 안정화)
- 로그인 후 사용자 멤버십 기준 churchSlug로 진입
- `/app` 엔트리에서 사용자 상태별 분기
- `/app/[churchSlug]/*` 접근 시 membership 가드 적용
- 권한 없음/워크스페이스 없음 상태에서 안내+다음 행동 제공

## E. 브랜딩 정렬
- 데모 교회명 통일:
  - name: `대흥교회 이음두빛`
  - slug: `daehung-ieum-dubit`
- 홈/앱/워크스페이스 문구를 “숨 플랫폼 > 교회 워크스페이스”로 정렬

## F. 계정 경로 분리
- 플랫폼 관리자 전용 경로 분리: `/platform-admin`
- 워크스페이스 사용자와 관리자 로그인 분기
- 홈에서 데모 워크스페이스 직접 유도 제거 (로그인 기반 진입)

## G. 공홈 리디자인 (브랜드형 랜딩)
- 카드 나열형 SaaS 소개에서 탈피
- 다크톤, 대형 타이포, 절제된 섹션 중심으로 재설계
- 구성: 미니멀 헤더 → 강한 히어로 → 브랜드 메시지 → 플랫폼 구조 → 모듈 소개 → CTA

## H. 홈 세션 유지 UX
- 로그인 상태에서 홈 진입 시 상태별 CTA 분기
  - 관리자: 플랫폼 콘솔 이동
  - 사용자: 내 워크스페이스 이동
  - 비로그인: 시작/로그인

## 4) 핵심 결정 사항 (Arti가 반드시 알고 있어야 할 원칙)

1. **완료 기준**
- 작업 완료 = 코드 반영 + 빌드 확인 + 커밋 + 푸시 (+ 가능하면 배포)

2. **브랜드/제품 원칙**
- 숨은 플랫폼 브랜드
- 교회는 플랫폼 내부 워크스페이스

3. **UX 원칙**
- 막힘 화면 금지: “권한 없음/워크스페이스 없음”은 안내 + 다음 행동 제공
- 미구현 기능은 활성처럼 보이지 않게 명시(준비 중/곧 연결)

4. **아키텍처 원칙**
- 신규 기능은 기본적으로 `/app/[churchSlug]` 기준
- 쿼리는 churchId 스코프를 기본 전제로 작성

5. **디자인 원칙**
- 공홈은 브랜드 랜딩 (관리자 화면 느낌 금지)
- 앱은 실무 운영 허브
- 모바일 사용성 절대 훼손 금지

## 5) 지금 남아있는 리스크 / 기술 부채

1. **레거시 단일 관리자 라우트 공존**
- `(admin)` 기반 옛 경로가 여전히 남아 있음
- 혼선 방지를 위해 리다이렉트 또는 내부 전용화 필요

2. **인증 단순화 상태**
- 현재는 임시 쿠키 기반
- 장기적으로 NextAuth/OIDC 또는 강화된 auth 계층 필요

3. **교회 생성/초대 실제 플로우 미완성**
- UI는 있으나 실제 생성/초대 수락 로직 미구현

4. **RBAC 미완성**
- OWNER/ADMIN/PASTOR/LEADER/VIEWER 세밀 권한 제어 미완

5. **빌드 시 DB 의존성 관리 주의**
- 환경변수(DATABASE_URL) 누락 시 오류 발생 가능
- 배포 파이프라인에서 env 안정화 필요

## 6) 재성 + Arti 협업 방식 제안 (실전용)

## 6-1. 역할 분담
- 재성: 구현 책임자(코드 품질/기술 의사결정)
- Arti: 작업 분해 + 코드 초안 + 테스트 체크리스트 + PR 설명 자동화
- 선용: 제품 방향/우선순위/브랜드 의사결정

## 6-2. 작업 단위 룰
- 1 작업 = 1 목표 + 1 완료 조건(DoD) + 1 커밋 묶음
- 예시 DoD:
  - 라우트 접근 가드 동작
  - 모바일 레이아웃 확인
  - 미구현 상태 표기 확인

## 6-3. 브랜치 전략(권장)
- `main`: 배포 가능 상태 유지
- `feat/*`: 기능 브랜치
- `fix/*`: 긴급 수정
- PR 제목 규칙 예시:
  - `feat(workspace): ...`
  - `fix(auth): ...`
  - `style(marketing): ...`

## 6-4. 커밋 메시지 규칙(권장)
- Conventional 형식 유지
- 예:
  - `feat(workspace): add onboarding create-church flow`
  - `fix(auth): enforce membership guard on workspace routes`
  - `style(marketing): refine hero typography and CTA hierarchy`

## 6-5. 리뷰 체크리스트
PR마다 아래 7개 체크:
1) 숨=플랫폼 / 교회=워크스페이스 위계가 유지되는가?
2) churchId 스코프 누락 쿼리가 없는가?
3) 미구현 기능이 활성처럼 보이지 않는가?
4) 모바일 레이아웃 깨짐이 없는가?
5) 접근권한 실패 시 안내+다음행동이 있는가?
6) 홈이 관리자 UI처럼 보이지 않는가?
7) 빌드/배포 환경변수 의존이 안전한가?

## 6-6. Arti에게 줄 작업 프롬프트 템플릿

```md
프로젝트: soom-church
작업 목표: (한 줄)
완료 조건(DoD):
- [ ]
- [ ]
제약:
- 숨=플랫폼, 교회=워크스페이스 유지
- churchId 스코프 강제
- 모바일 레이아웃 훼손 금지
출력:
- 변경 파일 목록
- 위험 포인트
- 테스트 시나리오
- rollback plan
```

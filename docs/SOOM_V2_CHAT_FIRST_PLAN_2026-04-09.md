# SOOM V2 CHAT-FIRST PLAN - 2026-04-09

## 제품 한 줄
숨은 목자가 목장 운영을 대화로 처리하고, 웹앱은 그 결과를 정리, 검토, 복구하는 운영 OS다.

## 이번 전환이 필요한 이유
1. 목자가 직접 폼을 많이 채워야 한다.
2. UI가 정보 정리보다 반복 노출이 많아 피곤하다.
3. 기존 관리툴과 본질적으로 다른 상품처럼 느껴지지 않는다.
4. `목자가 덜 헤매고 더 정확하게 사람을 관리하게 한다`는 목적 철학에서 멀어지고 있다.

## V1 / G.I.D.O 트랙에서 멈출 것
- 입력 중심 quick form 확장
- 화면마다 같은 이름을 반복 노출하는 구조
- 홈, 운영, 기록, 가정 편집이 한 화면에서 섞이는 구조
- 웹앱 안에서 목자가 대부분을 직접 쓰는 흐름

## V2 핵심 가설
목장 운영의 실제 입력은 폼이 아니라 대화에서 나온다.

그래서 숨의 기본 구조는 아래로 바뀐다.
- 입력 채널: 모라와의 채팅
- 해석 레이어: AI가 사람, 가정, 관계, 기도, 심방, 출석, 이벤트, 후속조치를 구조화
- 검토 레이어: 애매한 내용만 목자가 확인
- 기록 레이어: 웹앱은 정리된 결과를 보여주고 수정하는 운영판

## 채팅 우선 운영 루프
1. 목자가 텔레그램 같은 채팅에서 모라에게 말한다.
2. 모라가 내용을 목장 운영 데이터로 해석한다.
3. 확실한 내용은 기록 후보로 정리한다.
4. 애매한 내용만 다시 묻거나 `검토 대기`로 보낸다.
5. 웹앱에서는 오늘 변화, 검토 대기, 사람 상태, 가정 관계를 본다.
6. 목자는 필요한 부분만 승인하거나 수정한다.

## 웹앱의 새 역할
### 1. Today / Inbox
- 오늘 새로 들어온 변화
- 관리 필요 인원
- 검토 대기 항목
- 오늘 바로 처리할 3~5개

### 2. Review Queue
- AI가 추출한 기록 후보
- 누구 이야기인지 불확실한 항목
- 관계 연결이 애매한 항목
- 승인 / 수정 / 보류

### 3. People
- 사람 중심 레코드
- 이름, 상태, 최근 변화, 다음 액션
- 전체 폼이 아니라 요약 + 필요한 수정

### 4. Households / Relationships
- 가정 구조
- 배우자 / 자녀 / 가족 관계
- 가정 단위 중보와 메모
- 반복 이름 나열이 아니라 관계 중심 보기

### 5. Timeline / Records
- 심방
- 상담
- 출석
- 교회 이벤트
- 근황
- 후속 기록

### 6. Search / Recovery
- 이름, 가정, 메모, 최근 사건 검색
- `누가 최근 아팠지?`, `지난주 심방 누구였지?` 같은 복구

## 웹앱이 더 이상 아니어야 하는 것
- 거대한 입력 폼 묶음
- 설명이 많은 튜토리얼형 UI
- 이름과 가정명이 곳곳에 중복되는 리스트
- 목자가 데이터를 일일이 손으로 적는 어드민 패널

## UX 원칙
1. 이름은 한 화면에서 한 번 크게, 나머지는 최소화
2. 정보는 요약 -> 펼침 구조로 보여준다
3. 사람이 먼저, 폼은 나중
4. 대화에서 나온 변화가 바로 보이게 한다
5. `지금 무엇을 해야 하는지`가 항상 드러나야 한다
6. 디자인은 조용하고 선명해야 한다. 화려함보다 정리감이 우선이다

## AI / 모델 전략
### 바로 시작할 방식
- 1차 검증은 현재 모라 + GPT-5.4로 간다
- 이유: 이미 연결되어 있고, 바로 실험 가능하고, 제품 가설 검증이 먼저다

### 나중에 검토할 방식
- 숨 전용 에이전트 Gemma 4 또는 별도 모델
- 조건: 추출 스키마, 검토 흐름, 실제 운영 패턴이 안정된 뒤

## MVP 핵심 범위
### 포함
- 채팅 입력을 운영 데이터 후보로 변환
- 검토 대기 큐
- 사람 / 가정 / 관계 / 기록 레코드
- 오늘 관리 보드
- 검색 / 복구

### 제외
- 과한 커스텀 대시보드
- 보기용 카드 과다 생산
- 폼 우선 상세 CRUD 확장
- 모바일/PC 각각 다른 복잡한 정보 구조

## 즉시 실행 순서
1. G.I.D.O 입력형 확장 중단, 방향 전환 문서 고정
2. `chat -> extracted update -> review -> record` 흐름 스키마 설계
3. 웹앱 IA를 `Today / Review / People / Households / Timeline / Search`로 재설계
4. 첫 구현은 `Review Queue`부터 시작
5. 그 다음 `People` / `Households`는 정리판으로 다시 만든다
6. 채팅 입력은 우선 텔레그램의 모라를 그대로 사용한다

## 첫 화면 시안
### 홈
- 오늘 새로 들어온 변화
- 검토 대기
- 바로 연락할 사람
- 최근 기록된 가정 변화

### 검토 대기 카드 예시
- `김민수 어머니 수술 예정 -> 건강 기록으로 저장할까?`
- `이수진이 새가정 A의 배우자인지 확인 필요`
- `박지훈 2주 연속 결석 -> 관리 필요로 올릴까?`

## 성공 판단 기준
1. 목자가 입력보다 대화로 운영하게 되는가
2. 웹앱이 쓰는 곳이 아니라 확인하는 곳으로 바뀌는가
3. 같은 내용을 반복해서 보지 않아도 되는가
4. 기존 교회 관리툴과 다른 감각이 생기는가
5. 목자의 인지 마찰이 실제로 줄어드는가

## 현재 결정
- 숨의 다음 버전은 `chat-first`다
- 웹앱은 입력툴이 아니라 운영판이다
- 기존 G.I.D.O 화면은 참고물로만 남기고, 제품 기준은 여기서 다시 시작한다

# SOOM Visualization MVP Feasibility — 2026-03-26

## 1. 문서 목적

이 문서는 숨의 초기 차별점으로 떠오른
**소형 교회용 교인 구조 시각화 워크스페이스**가
실제로 MVP 수준에서 가능한지,
그리고 비용 측면에서 어떤 선택이 현실적인지 정리한 문서다.

초기 검토 기준은 아래 3가지다.

1. 기술적으로 지금 만들 수 있는가
2. AI 없이도 핵심 가치를 줄 수 있는가
3. 초기 운영비가 감당 가능한가

---

## 2. 결론 요약

결론부터 말하면,
**가능하다.**
그리고 초기 MVP는 **AI 없이 시작하는 것이 맞다.**

숨의 첫 MVP는 다음 정의로 보는 것이 가장 현실적이다.

> **100명 규모 이하 소형 교회를 대상으로,
> 교인 구조와 상태를 한눈에 보여주는 시각화 워크스페이스**

핵심은 화려한 3D가 아니라,
**실제로 운영에 도움이 되는 시각화 구조**를 먼저 만드는 데 있다.

---

## 3. 왜 이 MVP가 필요한가

소형 교회/개척교회 현장의 페인 포인트는 대체로 아래와 같다.

- 기존 교회 관리 프로그램은 다루기 어렵다.
- 기능은 많아도 한눈에 보이지 않는다.
- 누구를 챙겨야 하는지 바로 파악하기 어렵다.
- 사람, 가족, 교구, 목장, 심방, 공지가 서로 연결되어 보이지 않는다.
- 노션 같은 범용 도구는 자유롭지만 직접 구조를 설계해야 해서 곧 복잡해진다.

즉,
문제는 단순히 “데이터를 저장 못한다”가 아니라,
**운영 구조가 눈에 안 들어온다**는 것이다.

숨은 이 지점에서 차별화될 수 있다.

---

## 4. MVP 한 줄 정의

초기 MVP는 이렇게 정의한다.

### 제품 정의
**작은 교회를 위해 교인 구조와 상태를 한눈에 보여주는 운영 워크스페이스**

### 사용자 가치
- 지금 누구를 챙겨야 하는지 보인다.
- 가족/교구/목장 구조가 한 화면에 정리된다.
- 심방/후속/상태 변화가 흩어지지 않는다.
- 기록과 운영 액션이 연결된다.

---

## 5. MVP 범위

## 5.1 반드시 포함할 것

### A. 교인 구조 시각화
- 사람 목록이 아니라 구조 기반 보기
- 가족 단위 묶음
- 교구 / 목장 단위 그룹화
- 상태 색상 표시
- 후속 필요 여부 표시

### B. 교인 상세
- 기본 정보
- 현재 상태
- 최근 기록
- 심방기록 / 사역기록 / 메모
- 다음 액션

### C. 기록 연결
- 심방기록
- 상담 메모
- 사역 메모
- 사람과 연결된 운영 기록

### D. 최소 공지 흐름
- 공지 작성
- 대상 지정
- 최근 공지 확인

---

## 5.2 후순위로 미룰 것

### AI 기능
- 요약
- 자동 추천
- 자동 분류
- 초안 생성

### 고도화 시각화
- 완전한 3D 교적 공간
- 복잡한 애니메이션
- 고급 분석 대시보드

### 고급 운영 기능
- 정교한 권한 시스템
- 외부 채널 자동화
- 회계/헌금 고도화
- 복잡한 보고서 생성

## 6. AI 없이 가능한가

가능하다.
오히려 **초기 MVP는 AI 없이 가는 것이 더 적절하다.**

이유는 단순하다.

1. 초기 핵심 가치는 “예측”이 아니라 “가시화”다.
2. 지금 필요한 것은 지능형 추천보다 데이터 구조와 시각화다.
3. AI는 데이터가 쌓여야 가치가 커진다.
4. AI를 먼저 넣으면 제품 정체성이 흐려질 수 있다.

### AI 없이도 가능한 핵심 기능
- 교인 구조 시각화
- 가족/교구/목장 연결
- 상태 시스템
- 후속 필요 표시
- 기록 연결
- 최소 공지 흐름

즉,
초기 MVP는 **일반 웹앱 기술만으로 충분히 구현 가능하다.**

---

## 7.2 시각화 스택

### 1차 추천
**React Flow**

이유:
- 노드/연결 구조 표현이 빠르다.
- 사람 → 가족 → 목장 → 교구 흐름을 보여주기 쉽다.
- MVP 제작 속도가 빠르다.
- 운영용 구조 시각화에 적합하다.

### 2차 후보
**D3.js**

이유:
- 더 자유로운 커스텀 시각화 가능
- 하지만 개발 난이도가 더 높다

### 3차 실험 후보
**Three.js**

이유:
- 차별화 포인트로는 강함
- 그러나 MVP 핵심은 아님
- 잘못하면 장식에 비해 운영 가치가 약할 수 있음

### 추천 순서
1. React Flow
2. 필요 시 D3.js
3. 이후 실험으로 Three.js

추가로 중요한 것은
“새 테이블을 계속 늘리는 것”보다,
**시각화에 맞는 조회 구조와 상태 표현 방식**을 정리하는 것이다.

## 9. 시각화 MVP의 권장 구조

### 1차 구조 기준
**교구/목장 안에 가족과 개인이 들어가는 구조**

이유:
- 실제 교회 운영 방식과 잘 맞는다.
- 소형 교회에서도 이해가 쉽다.
- 큰 교회로 확장할 때도 구조를 유지하기 좋다.

### 화면에서 보여줘야 할 최소 요소
- 교구
- 목장
- 가족
- 개인
- 상태
- 후속 필요 여부

### 시각화에서 바로 보이면 좋은 상태 예시
- 새가족
- 정착중
- 심방 필요
- 후속 필요
- 안정
- 휴면

## 10. 비용 측면 검토

## 10.1 Google 로그인 비용
Google OAuth 자체는 일반적으로 **초기 비용 없이 시작 가능**하다.
즉,
로그인 기능만 붙이는 것 때문에 바로 큰 비용이 발생하지는 않는다.

돈이 들어가는 핵심 포인트는 보통 아래다.

- 서버/호스팅
- DB
- 파일 저장
- 이메일/문자 발송
- AI 기능
- 사용자 증가에 따른 인프라 사용량

---

## 10.2 MVP 단계에서의 현실적인 비용 구조

### 거의 필수 비용
- 도메인
- 배포 환경
- Postgres DB

### 선택적 비용
- 파일 스토리지
- 모니터링 도구
- 이메일 발송
- SMS / 카카오 알림
- AI API

### 초반 특징
초기 MVP 단계에서는
비용 부담의 중심이 인증이 아니라,
**호스팅과 DB 유지** 쪽에 있다.

즉,
Google 로그인 도입 여부가 아니라,
전체 서비스 운영 규모가 비용에 더 큰 영향을 준다.

---

## 10.3 비용 관점 결론

MVP는 **충분히 저비용으로 시작 가능**하다.
특히 아래 원칙을 지키면 부담을 크게 줄일 수 있다.

1. AI를 나중으로 미룬다.
2. 3D를 나중으로 미룬다.
3. React Flow 수준의 시각화부터 만든다.
4. 알림/SMS/외부 발송은 후순위로 둔다.
5. 소형 교회 100명 규모를 기준으로 최적화한다.

## 12. 최종 판단

숨의 초기 MVP는
기술적으로도 가능하고,
비용 측면에서도 충분히 현실적이다.

다만 성공의 핵심은
“AI를 얼마나 넣느냐”가 아니라,
**작은 교회가 사람 구조와 상태를 한눈에 볼 수 있게 하느냐**에 있다.

즉,
초기 제품은

> **100명 규모 소형 교회를 대상으로,
> 사람·가족·교구·목장 구조와 상태를 시각적으로 보여주는 운영 워크스페이스**

로 정의하는 것이 가장 적절하다.

이 정의 안에서 개발하면,
숨은 단순 관리툴이 아니라
작은 교회에 실제로 필요한 구조적 도구로 자리잡을 수 있다.

# SOOM Information Architecture Map v1

## 1. 문서 목적

이 문서는 `SOOM_PRODUCT_STRATEGY_2026.md`를 실제 제품 구조로 고정하기 위한 첫 번째 정보 구조 맵이다.

목표는 3가지다.

1. 핵심 모듈의 경계를 명확히 한다.
2. 페이지 구조와 엔티티 연결을 정리한다.
3. 이후 개발 시 화면 단위가 아니라 데이터 흐름 단위로 판단할 수 있게 만든다.

---

## 2. 제품 최상위 구조

숨의 최상위 구조는 아래 6개 모듈로 유지한다.

- 사람
- 공지 / 전달
- 작업
- 기록
- 콘텐츠 / 행사
- 설정

이 구조는 쉽게 바꾸지 않는다.
각 모듈은 메뉴 이름이 아니라 운영 흐름의 기준 축이다.

---

## 3. 모듈별 역할 정의

### 3.1 사람
역할:
- 교회 안의 사람 흐름을 관리하는 중심 모듈
- 등록, 상태, 후속관리, 배정, 연결의 시작점

포함 범위:
- 새가족 등록
- 인물 목록
- 상태 관리
- 담당자 배정
- 후속 메모
- 관련 작업 / 기록 / 공지 연결

핵심 질문:
- 지금 이 사람은 어떤 상태인가?
- 다음에 누가 무엇을 해야 하는가?
- 이 사람과 관련해 남아 있는 기록은 무엇인가?

### 3.2 공지 / 전달
역할:
- 전달할 내용을 만들고, 대상과 시점을 정하는 모듈
- 단순 게시판이 아니라 운영의 전달 허브

포함 범위:
- 공지 작성
- 대상 그룹 선택
- 예약 발송
- 전달 이력 확인
- 관련 행사 / 작업 / 사람 연결

핵심 질문:
- 무엇을 누구에게 보내야 하는가?
- 언제 전달되는가?
- 전달 이후 어떤 운영 액션이 이어지는가?

### 3.3 작업
역할:
- 실제 해야 할 일을 정리하고 handoff하는 모듈
- 운영의 말단이 아니라 실행 전환 지점

포함 범위:
- 작업 생성
- 담당자 지정
- 마감일 관리
- 우선순위 정리
- 사람 / 공지 / 콘텐츠 요청 기반 작업 생성

핵심 질문:
- 지금 해야 할 일은 무엇인가?
- 누가 맡았는가?
- 이 작업은 어떤 흐름에서 생겼는가?

### 3.4 기록
역할:
- 운영 중 발생한 메모, 상담, 결정, 이력을 남기는 모듈
- 보관이 아니라 다음 액션을 가능하게 하는 운영 로그

포함 범위:
- 사람별 기록
- 사역/팀 기록
- 상담 메모
- 상태 변경 로그
- 검색 가능한 운영 히스토리

핵심 질문:
- 이전에 무슨 일이 있었는가?
- 왜 이런 상태가 되었는가?
- 다음 담당자가 이해할 수 있는 맥락이 남아 있는가?

### 3.5 콘텐츠 / 행사
역할:
- 제작 요청, 행사 준비, 배포 연결을 다루는 모듈
- 숨의 차별점인 운영과 콘텐츠 handoff를 보여주는 축

포함 범위:
- 콘텐츠 요청 접수
- 검토 상태 관리
- 행사 준비 요청
- 제작/배포 연결
- 요청과 결과물 이력 관리

핵심 질문:
- 어떤 요청이 들어왔는가?
- 지금 어느 단계에 있는가?
- 누구의 확인 또는 실행이 필요한가?

### 3.6 설정
역할:
- 조직, 권한, 상태값, 기본 운영 규칙을 관리하는 모듈

포함 범위:
- 조직 구조
- 팀/부서 관리
- 사용자 권한
- 상태값 정의
- 전달 채널 설정
- 운영 템플릿 관리

핵심 질문:
- 시스템의 기본 규칙은 어떻게 정해지는가?
- 누가 무엇을 볼 수 있고 변경할 수 있는가?

## 4. 핵심 엔티티 맵

### 4.1 Person / Member
설명:
- 숨의 첫 진입 핵심축
- 새가족, 성도, 봉사자, 리더 등 운영 대상의 기본 단위

연결:
- Status
- Note / Record
- Task
- Notice 수신 대상
- Application
- OrganizationUnit

### 4.2 Status
설명:
- 사람, 요청, 작업, 공지 등 각 흐름의 현재 상태값
- 단순 라벨이 아니라 handoff 기준

예시:
- 새가족 등록됨
- 첫 연락 대기
- 정착 진행 중
- 담당자 배정 완료
- 응답 대기

연결:
- Person / Member
- Task
- Application
- ContentRequest / StudioRequest

### 4.3 Note / Record
설명:
- 상담, 메모, 활동 이력, 운영 판단 기록

연결:
- Person / Member
- Task
- Notice
- ContentRequest / StudioRequest

### 4.4 Task
설명:
- 실행을 위한 단위 작업

연결:
- Person / Member에서 생성 가능
- Notice 후속 작업으로 생성 가능
- ContentRequest 검토/제작 단계에서 생성 가능
- 담당자(User)와 연결

### 4.5 Notice
설명:
- 공지 / 전달 단위
- 내용, 대상, 시점, 상태를 가진다

연결:
- 대상 그룹 또는 Person 목록
- 행사/콘텐츠와 연결 가능
- 전달 후 Task 생성 가능

### 4.6 Application
설명:
- 신청, 등록, 응답 폼 기반 데이터
- 새가족 등록이나 사역 신청의 진입점

연결:
- Person 생성 또는 기존 Person 갱신
- Status 시작점
- 담당 후속 Task 생성 가능

### 4.7 ContentRequest / StudioRequest
설명:
- 콘텐츠 제작 또는 실행 요청 단위

연결:
- 요청자(Person/User)
- 관련 행사
- Task
- Record
- 배포용 Notice

### 4.8 OrganizationUnit
설명:
- 교구, 부서, 팀, 사역 단위 등 조직 구조

연결:
- Person 소속
- Notice 대상 지정
- Task 담당 범위
- 권한 및 운영 단위

## 6. 핵심 handoff 구조

숨의 핵심은 각 모듈이 따로 존재하는 것이 아니라,
운영 흐름이 다음 단계로 자연스럽게 이어지는 것이다.

### 핵심 handoff A
Application → Person → Status → Task → Record

예:
새가족 등록이 들어오면
사람이 생성되고,
첫 연락 대기 상태가 설정되고,
담당 작업이 생성되고,
결과가 기록으로 남는다.

### 핵심 handoff B
Notice → Target Selection → Scheduled Delivery → Response / Follow-up Task

예:
행사 공지를 작성하고,
청년부 대상에게 예약 발송하고,
이후 응답 확인 또는 현장 준비 작업으로 이어진다.

### 핵심 handoff C
ContentRequest → Review → Task → Distribution / Notice

예:
주보용 이미지 요청이 들어오면
검토 후 제작 작업이 생성되고,
완료되면 공지나 배포 흐름과 연결된다.

## 7. 대시보드 원칙

대시보드는 모든 것을 다 보여주는 화면이 아니라,
운영자가 지금 바로 움직여야 할 흐름을 보여주는 화면이어야 한다.

우선 노출할 블록:
- 오늘 처리할 새가족 / 사람 흐름
- 예정된 공지 / 전달
- 지연된 작업
- 최근 기록
- 대기 중인 콘텐츠 요청

즉,
대시보드는 요약판이고,
실제 액션은 각 모듈 상세로 들어가서 하게 만든다.

---

## 8. 설계 결정 원칙

1. 화면 추가보다 엔티티 연결을 먼저 본다.
2. 상세 화면은 반드시 다음 액션 진입점을 포함한다.
3. 상태값은 보기용이 아니라 handoff 기준으로 설계한다.
4. 하나의 정보는 하나의 기준 위치를 가진다.
5. 검색과 기록 복구가 가능한 구조를 유지한다.

# Feature Specs

- Last updated: 2026-03-12
- Source of truth for product scope: `../PRD.md`

## 목적

이 문서는 PRD를 실제 구현 단위로 분해한 기능 상세 문서다. Ralph loop 기준으로는 이 문서를 바탕으로 작업 단위를 자르고, acceptance criteria를 검증한다.

## 제품 상태 기준

- 현재 런칭 후보 범위는 관리자 운영 허브다.
- `core-service`가 대부분의 사용자 기능을 제공한다.
- `application-service`는 신청 도메인의 첫 번째 독립 서비스다.
- 콘텐츠/알림/AI/커뮤니티는 미래 모듈이며 현재 런칭 범위에서 제외한다.

## Feature 1. Authentication and Session

### 목적

운영자만 관리자 기능에 접근할 수 있어야 한다.

### 현재 범위

- 로그인
- 로그아웃
- 인증된 관리자 화면 접근 제어
- 플랫폼 관리자 전용 라우트 분리

### 주요 화면

- `/login`
- `/app`
- `/platform-admin`

### Acceptance Criteria

- 유효한 계정으로 로그인할 수 있다.
- 로그인 후 관리자 화면으로 이동한다.
- 로그아웃 후 보호된 화면에 직접 접근할 수 없다.
- 플랫폼 관리자 계정은 플랫폼 관리 화면에 접근할 수 있다.

## Feature 2. Admin Dashboard

### 목적

관리자가 하루의 운영 상태를 첫 화면에서 파악하고 바로 행동할 수 있어야 한다.

### 현재 범위

- 핵심 KPI 카드
- 최근 등록 교인
- 최근 신청
- 최근 공지
- 빠른 작업 진입

### 핵심 지표

- 총 교인 수
- 이번 달 신규 등록 수
- 미처리 신청 수
- 후속관리 필요 교인 수
- 교구별 인원 분포

### Acceptance Criteria

- 대시보드에 핵심 지표가 표시된다.
- 최근 데이터 영역이 비어 있어도 화면이 깨지지 않는다.
- 대시보드에서 주요 작업 화면으로 이동할 수 있다.

## Feature 3. Member Management

### 목적

교인 정보를 중앙에서 관리하고 후속관리의 기준 레코드로 사용한다.

### 현재 범위

- 교인 목록 조회
- 교인 상세 조회
- 교인 등록
- 교인 수정
- 소프트 삭제 지원

### 주요 필드

- 이름
- 성별
- 생년월일
- 연락처
- 이메일
- 주소
- household 연결
- district 연결
- group 연결
- 등록일
- 직분
- 상태 태그
- 후속관리 필요 여부
- 메모

### Acceptance Criteria

- 관리자가 교인을 생성할 수 있다.
- 관리자가 기존 교인 정보를 수정할 수 있다.
- 목록과 상세 화면에서 핵심 필드를 확인할 수 있다.
- 삭제된 교인은 무결성을 깨지 않고 관리된다.

## Feature 4. Follow-Up Workflow

### 목적

새가족과 관리 필요 대상을 상태 기반으로 추적할 수 있어야 한다.

### 현재 범위

- 상태 태그 기반 분류
- `requiresFollowUp` 기반 추적
- 대시보드에서 후속관리 필요 수치 노출

### 상태 태그 예시

- 등록대기
- 새가족
- 정착중
- 목장배정완료
- 봉사연결
- 휴면
- 심방필요

### Acceptance Criteria

- 교인별 상태 태그가 저장된다.
- 후속관리 필요 여부를 식별할 수 있다.
- 대시보드 수치와 교인 데이터가 일관된다.

## Feature 5. District and Group Management

### 목적

교구와 목장 구조 안에서 교인 배정을 유지할 수 있어야 한다.

### 현재 범위

- 교구 조회
- 목장 조회
- 교인과 교구/목장 연결
- 리더 이름 관리의 기본 형태

### Acceptance Criteria

- 교구와 목장이 목록 또는 관리 화면에서 보인다.
- 교인을 교구/목장에 연결할 수 있다.
- 배정 변경이 교인 상세에 반영된다.

## Feature 6. Application Management

### 목적

교회의 신청 흐름을 한 곳에서 받고 상태를 관리한다.

### 현재 범위

- 신청 폼 조회/생성
- 신청 접수
- 신청 목록 조회
- 신청 요약 조회
- 신청 상태 변경

### 서비스 경계

- UI 셸: `core-service`
- API 및 데이터 계약: `application-service`

### Acceptance Criteria

- 신청 폼을 조회하고 생성할 수 있다.
- 신청을 등록할 수 있다.
- 신청 목록을 조회할 수 있다.
- 신청 상태를 `PENDING`, `IN_REVIEW`, `APPROVED`, `REJECTED`로 변경할 수 있다.
- 대시보드 또는 신청 화면에서 미처리 수를 확인할 수 있다.

## Feature 7. Notice Management

### 목적

운영 공지를 생성하고 최근 공지를 관리자 화면에 노출한다.

### 현재 범위

- 공지 목록
- 공지 생성/관리
- 고정 공지 여부
- 최근 공지 노출

### Acceptance Criteria

- 공지를 생성할 수 있다.
- 공지 목록에서 최근 공지를 확인할 수 있다.
- 대시보드 최근 공지 섹션과 관리 화면 데이터가 일치한다.

## Feature 8. Activity Logs

### 목적

운영 변경 이력을 검토할 수 있어야 한다.

### 현재 범위

- 활동 로그 목록 또는 요약 조회
- actor, action, target 정보 보관

### Acceptance Criteria

- 활동 로그 화면에서 최근 이력을 확인할 수 있다.
- 누가 어떤 대상에 어떤 작업을 했는지 최소 정보가 보인다.

## Feature 9. Platform Admin

### 목적

여러 교회 워크스페이스를 관리하는 플랫폼 운영 기능을 제공한다.

### 현재 범위

- 플랫폼 관리자 대시보드
- 교회 목록
- 구독 정보
- 워크스페이스 생성 진입

### Acceptance Criteria

- 플랫폼 관리자만 해당 화면에 접근할 수 있다.
- 교회/구독 개요를 조회할 수 있다.

## Feature 10. Multi-Tenant Foundation

### 목적

교회별 데이터 격리를 제품 구조에 반영한다.

### 현재 범위

- `Church`
- `ChurchMembership`
- `Subscription`
- 주요 엔티티의 `churchId` 스코프

### Acceptance Criteria

- 핵심 운영 엔티티에 교회 범위가 존재한다.
- 사용자와 교회의 관계를 표현할 수 있다.
- 구독 상태를 저장할 수 있다.

## Out of Scope for Launch

- 실시간 채팅
- 게시판/피드 기반 커뮤니티
- 설교 업로드/요약/쇼츠/TTS
- 다채널 알림 운영 자동화
- 범용 AI 오케스트레이션
- 성도용 공개 포털

## Open Execution Gaps

- 권한 모델은 현재 관리자 중심으로 단순하다.
- 교인/공지/활동 로그의 외부 공개 API 계약은 아직 정식 문서화되어 있지 않다.
- 신청 도메인은 서비스 분리 중간 상태라 폴백 제거 계획이 필요하다.
- 런칭 전 QA 시나리오와 시드 데이터 검증이 추가로 필요하다.

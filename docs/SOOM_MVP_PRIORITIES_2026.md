# SOOM MVP Priorities 2026

## 1. 문서 목적

이 문서는 `SOOM_PRODUCT_STRATEGY_2026.md`와 관련 실행 문서를 기준으로,
지금 만들 기능과 뒤로 미룰 기능을 구분하기 위한 MVP 우선순위 문서다.

핵심 기준은 화려함이 아니라 아래 3가지다.

1. 실제 교회 운영 흐름을 끊지 않는가
2. 반복 업무를 줄이는가
3. 다음 handoff를 자연스럽게 만드는가

---

## 2. MVP의 한 줄 정의

MVP는
**교회가 사람 흐름을 놓치지 않고,
기록과 작업과 전달을 한 구조 안에서 이어갈 수 있게 만드는 첫 운영 워크스페이스**다.

즉,
모든 기능을 다 넣는 버전이 아니라,
가장 반복적이고 중요한 운영 흐름을 실제로 돌릴 수 있는 버전이어야 한다.

---

## 3. 지금 당장 만들 기능

## 3.1 최우선 영역: 사람 흐름

### 반드시 포함
- 새가족 등록 입력
- 사람 목록
- 사람 상세
- 상태값 관리
- 담당자 배정
- 기록 작성
- 후속 작업 생성
- 상태 변경 이력 확인

### 이유
- 첫 진입 핵심축이 사람 흐름이기 때문
- 가장 반복되고 중요한 운영 문제와 직접 연결되기 때문
- 숨의 가치가 가장 빨리 드러나는 영역이기 때문

---

## 3.2 두 번째 영역: 작업 연결

### 반드시 포함
- 작업 생성
- 담당자 지정
- 마감일 설정
- 상태 관리
- 사람/공지/요청과 작업 연결
- 작업 목록/보드

### 이유
- 운영에서 실제 실행은 결국 작업으로 떨어지기 때문
- 기록만 남고 실행이 끊기는 문제를 막아야 하기 때문

---

## 3.3 세 번째 영역: 기록 구조

### 반드시 포함
- 사람 관련 기록 작성
- 작업 관련 기록 작성
- 시간순 기록 확인
- 최소한의 검색 또는 상세 이력 확인

### 이유
- 숨의 차별점은 정보 보관이 아니라 맥락 복구 가능성에 있음
- 담당자가 바뀌어도 이어서 움직일 수 있어야 함

---

## 3.4 네 번째 영역: 공지 / 전달 기본형

### 반드시 포함
- 공지 작성
- 대상 지정
- 예약 상태 관리
- 전달 이력 저장
- 공지 기반 후속 작업 생성

### 이유
- 운영과 전달이 분리되면 숨의 구조적 장점이 약해짐
- 단, 실제 외부 채널 고도화보다 내부 흐름 정리가 먼저임

---

## 3.5 다섯 번째 영역: 콘텐츠 요청 기본형

### 반드시 포함
- 요청 접수
- 상태값 관리
- 요청 상세
- 관련 작업 연결
- 결과물 링크/메모 저장

### 이유
- 숨만의 차별점을 보여주는 축이기 때문
- 다만 사람 흐름과 공지 흐름이 먼저 안정화되어야 함

---

## 4. MVP에서 미뤄도 되는 기능

아래 기능은 중요할 수 있지만,
MVP 핵심 가치를 검증하는 데 필수는 아니다.

### 후순위 기능
- 복잡한 자동화 규칙 엔진
- 정교한 통계 대시보드
- 화려한 차트형 인사이트
- 다채널 외부 발송 고도화
- 세밀한 권한 커스터마이징
- 고급 템플릿 마켓
- 반복 공지/캠페인 자동 시퀀스
- 모바일 앱 전용 경험 최적화
- 고급 파일 관리 시스템
- 복잡한 캘린더 통합

### 미루는 이유
- 초기에는 운영 핵심 흐름이 먼저 살아야 함
- 복잡도 대비 학습 비용이 큼
- 잘못 넣으면 제품이 기능 모음처럼 보일 위험이 있음

---

## 5. MVP 검증 질문

기능을 추가하기 전에 아래 질문으로 먼저 검증한다.

1. 이 기능이 사람 흐름, 전달 흐름, 작업 흐름 중 하나를 실제로 더 낫게 만드는가?
2. 지금도 사람이 수기로 반복하는 일을 줄여주는가?
3. 다음 담당자가 맥락을 이어받는 데 도움이 되는가?
4. handoff가 더 명확해지는가?
5. 없어도 MVP의 핵심 가치는 성립하는가?

5번 질문에 “그렇다”면,
그 기능은 우선순위를 낮춘다.

---

## 6. 권장 구현 순서

## 1단계
사람 흐름 MVP
- 새가족 등록
- 사람 목록 / 상세
- 상태 관리
- 기록 작성
- 첫 연락 Task 연결

## 2단계
작업 + 기록 안정화
- 작업 보드
- 담당자 흐름
- 상태 전환 로그
- 사람/작업/기록 간 연결 강화

## 3단계
공지 / 전달 기본형
- 공지 작성
- 대상 선택
- 예약/전달 상태
- 후속 작업 연결

## 4단계
콘텐츠 요청 기본형
- 요청 접수
- 검토 상태
- 제작 작업 연결
- 결과물 / 배포 연결

---

## 7. 실사용 검증 포인트

### 검증 포인트 A. 새가족 흐름
질문:
- 등록 이후 누가 무엇을 해야 하는지 바로 보이는가?
- 첫 연락 누락이 줄어드는가?
- 담당자가 바뀌어도 맥락 복구가 되는가?

### 검증 포인트 B. 공지 / 전달 흐름
질문:
- 공지 대상과 시점이 명확한가?
- 전달 뒤 해야 할 준비가 빠지지 않는가?

### 검증 포인트 C. 콘텐츠 요청 흐름
질문:
- 요청이 흩어지지 않고 한곳에서 보이는가?
- 검토와 제작 단계가 추적 가능한가?

---

## 8. 제품 판단 기준

MVP에서 좋은 기능은 아래 특징을 가진다.

- 새 일을 늘리지 않는다.
- 같은 설명을 반복하지 않게 만든다.
- 현재 상태와 다음 액션을 분명히 보여준다.
- 기록을 남기되, 읽기 쉬운 운영 맥락으로 남긴다.
- 교회와 사역팀이 바로 이해할 수 있는 언어를 쓴다.

---

## 9. 당장 팀이 붙들어야 할 한 문장

지금 숨이 만들어야 하는 것은
많은 기능이 아니라,
**사람 → 작업 → 기록 → 전달로 이어지는 실제 운영 흐름이 끊기지 않는 첫 제품**이다.

이 기준을 벗어나는 기능은
좋아 보여도 MVP 우선순위에서 뒤로 보낸다.

# SOOM Core Workflows v1

## 1. 문서 목적

이 문서는 숨에서 먼저 완성해야 할 핵심 시나리오 3개를
실제 제품 흐름 단위로 정의한 문서다.

기준은 단순하다.
- 현장에서 바로 쓸 수 있어야 한다.
- 중간 handoff가 끊기지 않아야 한다.
- 기록과 후속 작업이 자동으로 이어질 수 있어야 한다.

---

## 2. 워크플로우 1
# 새가족 등록 → 첫 연락 → 정착 연결

## 2.1 목표
- 사람 흐름의 핵심 가치를 가장 먼저 증명한다.
- 등록 이후 후속관리 누락을 줄인다.
- 상태, 담당, 기록, 다음 액션이 한 구조 안에서 이어지게 만든다.

## 2.2 시작 조건
- 새가족 등록 폼이 제출됨
또는
- 운영자가 직접 새가족 정보를 입력함

## 2.3 종료 조건
아래 중 하나로 정리된다.
- 정착 연결 완료
- 후속관리 진행 중
- 연락 불가 / 보류
- 타 사역/부서로 handoff 완료

## 2.4 주요 엔티티
- Application
- Person
- Status
- Task
- Record
- OrganizationUnit

## 2.5 기본 단계

### 단계 1. 등록 접수
시스템 동작:
- Application 생성
- 기본 접수 시각 저장
- 새가족 소스/유입 경로 저장 가능

운영자에게 필요한 것:
- 누가 등록되었는지 바로 보인다.
- 누락 없이 처리 대기 큐에 들어간다.

### 단계 2. 사람 생성 또는 기존 사람 매칭
시스템 동작:
- 신규면 Person 생성
- 기존 데이터와 중복 가능성이 있으면 검토 표시
- Person에 초기 Status 부여

권장 초기 상태 예시:
- 등록됨
- 첫 연락 대기

### 단계 3. 담당자 배정 및 첫 연락 작업 생성
시스템 동작:
- 담당자 지정 가능
- 첫 연락 Task 자동 생성 가능
- 기한 설정 가능

핵심 포인트:
- 등록만 되고 멈추지 않게 한다.
- 반드시 다음 액션이 생기게 한다.

### 단계 4. 연락 시도 및 기록 남김
운영자 동작:
- 전화, 문자, 메시지, 대면 접촉 시도
- 결과를 Record로 남김

기록 예시:
- 연락 완료 / 미응답 / 재연락 필요
- 관심 분야
- 다음 방문 예정
- 연결이 필요한 부서

### 단계 5. 상태 전환
기록과 결과에 따라 상태를 바꾼다.

예시 상태 흐름:
- 등록됨
- 첫 연락 대기
- 첫 연락 완료
- 정착 진행 중
- 정착 연결 완료
- 연락 불가
- 보류

### 단계 6. 정착 또는 후속 handoff
가능한 후속 동작:
- 담당 사역자에게 연결
- 소그룹 / 부서 배정
- 환영 안내 전달
- 다음 만남 일정 Task 생성

### 단계 7. 흐름 마감 또는 지속 관리
완료 시:
- 정착 관련 상태를 마감 처리
- 이후 일반 성도 관리 흐름으로 편입

---

## 2.6 필수 화면 요소

### 사람 목록 / 새가족 큐에서 보여야 할 것
- 이름
- 등록일
- 현재 상태
- 담당자
- 다음 액션 기한

### 사람 상세에서 보여야 할 것
- 기본 정보
- 상태 타임라인
- 최근 기록
- 연결된 작업
- 조직/사역 연결

---

## 2.7 MVP에서 꼭 필요한 자동화
- 등록 시 Person 생성 또는 매칭 보조
- 첫 연락 Task 생성
- 상태 변경 이력 기록
- 기록 작성 후 다음 Task 생성 가능

---

## 2.8 성공 판단 기준
- 등록 후 첫 연락 누락률 감소
- 담당자/상태 파악 시간 단축
- 한 사람에 대한 맥락 복구 속도 향상

---

## 3. 워크플로우 2
# 공지 작성 → 대상 지정 → 예약 / 전달

## 3.1 목표
- 공지가 흩어지지 않게 만든다.
- 누가, 언제, 무엇을 받는지 명확하게 만든다.
- 전달 이후 필요한 후속 작업까지 연결한다.

## 3.2 시작 조건
- 운영자가 공지를 작성하려고 함
- 행사, 일정, 사역 전달 사항이 발생함

## 3.3 종료 조건
아래가 명확해지면 완료다.
- 공지가 전달됨
- 전달 대상과 시점이 기록됨
- 필요한 후속 작업이 연결됨

## 3.4 주요 엔티티
- Notice
- Person / Group / OrganizationUnit
- Status
- Task
- Record

## 3.5 기본 단계

### 단계 1. 공지 초안 작성
입력 항목 예시:
- 제목
- 본문
- 전달 목적
- 관련 행사/콘텐츠

### 단계 2. 대상 지정
대상 방식 예시:
- 전체
- 특정 부서
- 특정 사역팀
- 특정 상태의 사람들
- 직접 선택한 사람 목록

핵심 포인트:
- 대상 지정은 운영 구조와 연결되어야 한다.
- 임시 복붙 발송이 아니라 기준 있는 타겟팅이 가능해야 한다.

### 단계 3. 전달 시점 선택
선택지 예시:
- 즉시 전달
- 예약 전달
- 반복 전달은 후순위

필수 정보:
- 언제 보낼지
- 누가 승인/작성했는지
- 현재 상태가 무엇인지

### 단계 4. 전달 실행 및 이력 저장
시스템 동작:
- Notice 상태 업데이트
- 전달 시각 기록
- 대상 정보 저장

예시 상태:
- 초안
- 예약됨
- 전달 완료
- 실패 / 보류

### 단계 5. 후속 작업 연결
예시:
- 행사 준비 Task 생성
- 응답자 확인 Task 생성
- 현장 안내 담당 Task 생성

---

## 3.6 필수 화면 요소

### 공지 목록에서 보여야 할 것
- 제목
- 대상
- 예약 시점
- 상태
- 작성자

### 공지 상세에서 보여야 할 것
- 내용
- 대상 조건
- 전달 상태
- 관련 작업
- 관련 행사/콘텐츠

---

## 3.7 MVP에서 꼭 필요한 자동화
- 대상 선택 구조
- 예약 상태 관리
- 전달 이력 저장
- 공지 기반 후속 Task 생성

---

## 3.8 성공 판단 기준
- 같은 공지 내용을 여러 채널에서 따로 관리하는 반복 감소
- 누가 받았는지 모호한 상황 감소
- 전달 후 준비 누락 감소

---

## 4. 워크플로우 3
# 콘텐츠 요청 → 검토 → 제작 / 배포 연결

## 4.1 목표
- 숨의 차별점인 운영과 콘텐츠 handoff를 보여준다.
- 요청이 메시지로 흩어지지 않게 한다.
- 검토와 실행을 기록 가능한 흐름으로 만든다.

## 4.2 시작 조건
- 사역팀이 콘텐츠 또는 행사 관련 요청을 등록함
- 운영자가 직접 요청을 정리해 입력함

## 4.3 종료 조건
아래 중 하나로 정리된다.
- 제작 완료 및 배포 연결 완료
- 요청 보류
- 요청 반려
- 추가 정보 요청 중

## 4.4 주요 엔티티
- ContentRequest / StudioRequest
- Status
- Task
- Record
- Notice
- Event(또는 행사 단위 연결 정보)

## 4.5 기본 단계

### 단계 1. 요청 접수
입력 항목 예시:
- 요청 제목
- 요청 목적
- 사용 일정
- 필요한 산출물
- 요청자
- 참고 링크/자료

### 단계 2. 검토
검토 포인트:
- 진짜 필요한 요청인지
- 일정이 가능한지
- 누가 승인해야 하는지
- 기존 작업/행사와 연결되는지

상태 예시:
- 접수됨
- 검토 중
- 추가 정보 필요
- 승인됨
- 반려됨

### 단계 3. 작업 분해 및 담당 배정
시스템/운영 동작:
- 제작 Task 생성
- 검수 Task 생성
- 배포 준비 Task 생성

핵심 포인트:
- 요청 하나가 실행 단위로 쪼개져야 한다.
- 사람이 기억하지 않아도 다음 단계가 보여야 한다.

### 단계 4. 진행 기록 남김
예시:
- 시안 업로드
- 피드백 반영
- 일정 변경
- 최종 승인

### 단계 5. 배포/전달 연결
가능한 연결:
- 행사 자료로 사용
- 공지에 첨부
- 별도 전달 채널로 배포

### 단계 6. 완료 처리
완료 시 남겨야 할 것:
- 결과물 링크
- 완료 일시
- 사용처
- 다음에 재사용 가능한 메모

---

## 4.6 필수 화면 요소

### 요청 목록에서 보여야 할 것
- 제목
- 요청자
- 사용 일정
- 상태
- 담당자

### 요청 상세에서 보여야 할 것
- 요청 내용
- 첨부 자료
- 상태 타임라인
- 연결된 작업
- 결과물 / 배포 이력

---

## 4.7 MVP에서 꼭 필요한 자동화
- 요청 접수 폼
- 상태 전환 기록
- 작업 생성 연결
- 결과물 링크 저장
- 공지/배포 연결 포인트

---

## 4.8 성공 판단 기준
- 콘텐츠 요청이 카톡/구두/메모로 흩어지는 문제 감소
- 요청 진행 상태 확인 시간 단축
- 제작과 배포 연결 누락 감소

---

## 5. 공통 설계 원칙

세 워크플로우 모두 아래 원칙을 공유한다.

1. 시작 이벤트가 명확해야 한다.
2. 현재 상태가 한눈에 보여야 한다.
3. 다음 액션이 비어 있으면 안 된다.
4. 기록이 남아 다음 담당자가 복구 가능해야 한다.
5. 각 흐름은 다른 모듈과 자연스럽게 연결되어야 한다.

---

## 6. 구현 우선순위 결론

가장 먼저 끝낼 흐름:
1. 새가족 등록 → 첫 연락 → 정착 연결

그 다음:
2. 공지 작성 → 대상 지정 → 예약/전달

그 다음:
3. 콘텐츠 요청 → 검토 → 제작/배포 연결

이 순서를 유지해야
숨이 사람 관리 도구로 축소되지 않으면서도,
가장 먼저 현장 효용을 증명할 수 있다.

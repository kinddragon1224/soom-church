# SOOM V2 PRODUCT STRUCTURE - 2026-04-10

## 한 줄 정의
숨은 `목자가 채팅앱에 목장 운영 내용을 간편하게 입력하면, 앱 내부 모델이 이를 자동 구조화해서 관리자 워크스페이스 데이터로 적재해주는 목장 운영 OS`다.

---

## 1. 제품 중심 재정의

### 우리가 만들고 있는 것
- 특정 `기도 목장` 전용 관리툴이 아니다.
- 여러 목자가 쓸 수 있는 `채팅 입력 기반 목장 운영 시스템`이다.
- `기도 목장`은 베타 레퍼런스 사용자 1곳이다.

### 핵심 원리
- 목자는 표, 폼, 여러 화면을 돌아다니며 입력하지 않는다.
- 목자는 그냥 채팅앱에 자연스럽게 적는다.
- 모델이 이를 읽고 아래 데이터로 자동 정리한다.
  - 사람 등록/보완
  - 심방 기록
  - 기도제목
  - 교회 이벤트
  - 출석 변화
  - 후속조치
  - 가족/가정 관계
- 관리자 워크스페이스에는 `정리된 데이터`가 쌓인다.
- 원문 채팅은 기본적으로 관리자 화면의 중심이 아니다.

---

## 2. 제품 구조

## A. 목자가 쓰는 것: 채팅 입력 인터페이스

### 역할
- 가장 쉬운 입력 채널
- 목자가 이미 쓰는 채팅 습관을 그대로 사용
- 반복 입력 피로를 줄임

### 입력 예시
- `오상준 형제 허벅지 다쳐서 내일 병원 간대. 톡방에 중보기도 올렸어.`
- `김민수 이번 주 예배 못 왔고 다음 주에 연락해봐야 할 것 같아.`
- `박지은 자매 오늘 처음 등록했어. 남편이랑 같이 왔어.`

### 채팅 인터페이스 원칙
- 자유 텍스트 입력
- 목장 관리 외 일반 대화 최소화
- 답변도 운영형으로 제한
  - 확인
  - 정리
  - 누락 질문
  - 후속 제안
- 잡담형 AI가 아니라 `목장 운영 입력 인터페이스`여야 함

---

## B. 시스템 내부: 비공개 구조화 파이프라인

### 역할
- 채팅 원문을 운영 데이터로 변환
- 사용자는 이 단계를 직접 보지 않아도 됨

### 파이프라인
1. capture
2. extract
3. classify
4. resolve
5. apply
6. audit

### 세부 설명
- capture: 채팅 입력 수집
- extract: 사람, 사건, 기도, 일정, 관계, 상태 변화 추출
- classify: 어떤 데이터 유형인지 판단
- resolve: 기존 사람/가정/관계와 연결
- apply: 실제 워크스페이스 데이터에 반영
- audit: 최소한의 시스템 로그만 유지

### 중요한 원칙
- 채팅 원문은 운영화면 메인이 아님
- 필요한 경우에만 내부 확인용으로 접근
- 기본 화면은 `정리된 결과` 중심

---

## C. 관리자가 보는 것: 관리자 워크스페이스

### 역할
- 목장 운영 데이터가 쌓이는 운영판
- 입력툴이 아니라 상태판, 관리판, 복구판

### 최상위 탭 제안
1. Overview
2. Members
3. Visits & Care
4. Events
5. Prayer
6. Follow-up
7. Households
8. Timeline
9. Settings

---

## 3. 탭별 역할

## 1) Overview
### 목적
- 지금 목장이 어떤 상태인지 한눈에 보기

### 보여줄 것
- 오늘 확인할 사람
- 최근 새로 등록된 사람
- 최근 심방/기도/이벤트 반영
- 후속 필요한 사람
- 누락 가능성이 큰 항목

### 하면 안 되는 것
- 복잡한 편집 기능 몰아넣기
- 긴 표 나열

---

## 2) Members
### 목적
- 사람 중심 관리의 메인 화면

### 반드시 유지할 것
- 목자가 직접 목원 추가 가능
- 목자가 직접 정보 수정 가능
- 목자가 직접 등록 가능

### 하지만 UI 원칙
- 복잡하면 안 됨
- 첫 화면은 표 하나 + 핵심 액션 몇 개만
- 고급 입력은 상세 화면으로 내림

### Members 목록에서 보여줄 것
- 이름
- 가정
- 상태
- 최근 기록 한 줄
- 후속 필요 여부

### 상단 핵심 액션
- `목원 추가`
- `빠른 등록`
- `최근 등록 보기`
- `후속 필요한 사람만 보기`

### 상세 화면에서 다룰 것
- 기본 정보
- 심방/돌봄 기록
- 기도제목
- 교회 이벤트
- 출석 변화
- 가족 관계
- 후속 이력

### UX 원칙
- 목록은 단순
- 상세는 풍부
- 한 화면에 모든 입력폼을 다 때려넣지 않음

---

## 3) Visits & Care
### 목적
- 심방, 상담, 건강, 돌봄 기록 관리

### 보여줄 것
- 최근 심방 기록
- 건강 이슈
- 상담/돌봄 메모
- 사람별 최근 케어 흐름

---

## 4) Events
### 목적
- 교회 이벤트와 개인 이벤트 관리

### 예시
- 등록
- 침례/세례
- 입교
- 성찬
- 헌아
- 중요한 일정/병원 방문

---

## 5) Prayer
### 목적
- 기도제목을 사람/가정 기준으로 관리

### 보여줄 것
- 현재 열린 기도제목
- 최근 추가된 기도
- 응답/종결 처리
- 가정별 기도제목

---

## 6) Follow-up
### 목적
- 연락, 확인, 챙김이 필요한 사람 관리

### 보여줄 것
- 오늘 연락할 사람
- 이번 주 확인 대상
- 일정 지난 후속조치

---

## 7) Households
### 목적
- 가정 단위 구조와 관계 관리

### 보여줄 것
- 가정 구성
- 배우자/자녀/가족 관계
- 가정 단위 기도/메모
- 가정 단위 최근 이슈

---

## 8) Timeline
### 목적
- 최근 운영 데이터 흐름 보기

### 보여줄 것
- 최근 반영된 등록
- 심방 기록
- 기도 추가
- 이벤트 반영
- 후속 생성

### 주의
- 원문 채팅이 아니라 `반영된 운영 로그` 중심

---

## 9) Settings
### 목적
- 운영 정책 관리

### 이후 다룰 것
- assistant 역할 제한
- 허용 답변 범위
- 자동 반영 강도
- 검토 필요 조건
- 개인정보/원문 보존 정책

---

## 4. Review를 어떻게 볼 것인가

### 기존 문제
- 현재 Review가 사용자 입장에서 또 하나의 입력 단계처럼 보일 수 있음
- 채팅앱 장점을 해칠 수 있음

### 새 기준
- Review는 메인 탭이 아니라 `예외 처리함`에 가깝다.
- 정말 애매한 것만 관리자에게 보인다.
- 이름도 `Review`보다 `확인 필요`가 더 적합할 수 있다.

### 언제만 띄울까
- 사람 매칭이 불명확함
- 같은 이름이 많음
- 관계가 모호함
- 기존 데이터와 충돌함
- 자동 반영 확신이 낮음

### 원칙
- 확실한 것은 자동 반영
- 애매한 것만 최소 개입 확인
- 관리자 워크스페이스 메인은 Review가 아니라 Members/Follow-up/Prayer/Timeline이어야 함

---

## 5. 목원 관리 UI 원칙

### 유지할 것
- 직접 추가
- 직접 수정
- 직접 등록

### 제거할 것
- 한 화면에 너무 많은 폼
- 같은 이름 반복 노출
- 운영/입력/설명/검토가 섞인 화면
- 표와 카드와 입력박스가 동시에 과하게 보이는 구조

### 추천 구조

#### Members 메인
- 검색
- 필터
- 사람 목록
- 상단 버튼 2~4개

#### Quick add 모달
- 이름
- 연락처
- 가정
- 기본 메모

#### 상세 화면
- 사람이력 중심
- 여기서 추가 수정 가능

즉,
- `목록은 단순`
- `입력은 짧게`
- `상세는 깊게`

---

## 6. assistant 역할 제한 원칙

### 원하는 방향
assistant는 일반 AI 비서가 아니라 `목장 운영 전용 입력 인터페이스`여야 한다.

### 해야 하는 것
- 사람 등록 내용 정리
- 심방 기록 정리
- 기도제목 정리
- 이벤트 정리
- 후속조치 제안
- 누락 정보 짧게 질문

### 하지 말아야 하는 것
- 일반 상식 잡담 길게 응답
- 창작형 수다
- 목장 관리와 무관한 범용 비서 역할

### 답변 톤 원칙
- 짧고 운영형
- 확인/정리 중심
- 필요할 때만 질문

예:
- `오상준 형제 건강 기록과 기도제목으로 정리할게. 내일 병원 방문 일정도 후속으로 둘까?`
- `김민수는 기존 등록 인물과 동일한지 확인이 필요해. 어느 김민수인지 알려줘.`

---

## 7. 바로 다음 제품 우선순위

1. 채팅 원문 중심 화면이 아니라 관리자 워크스페이스 중심 IA로 다시 정리
2. Members를 단순한 메인 운영 화면으로 재설계
3. 채팅 입력 결과가 Members / Prayer / Follow-up / Timeline에 자동 적재되게 연결
4. 예외 케이스만 `확인 필요`로 분리
5. assistant를 목장 관리 전용으로 제한하는 프롬프트/정책 설계

---

## 8. 최종 기준

숨은 `채팅앱처럼 입력하고, 관리자 워크스페이스처럼 관리하는 목장 운영 OS`다.

- 입력은 대화처럼 쉽고
- 결과는 구조화되어 쌓이고
- 관리 화면은 단순하고
- 직접 추가/수정 기능은 남기되
- UI는 복잡하지 않게 유지한다.
